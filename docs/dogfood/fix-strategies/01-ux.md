**Lens:** Experience design, onboarding, handholding, copy clarity, jurisdiction awareness, "feels nowhere" eradication

## Top 5 UX Bugs (Adoption-Impact Ranked)

### 1. **Project Spine wiring is only 3 of 17 workflows** ⚠️ DEMO-BLOCKER
**Impact:** 5 (Credibility instant-kill) | **Effort:** L (15-20min per workflow × 14)  
**Surface:** q6 (Job Sequencing), q8 (Permit Applications), q11 (Supply Ordering), q15 (Daily Log) — all unwired  
**The problem:** When John clicks "Permits" or "Daily Log," he hits a blank workflow with zero project context. No banner. No pre-fill. No peer-workflow nav. Instant credibility collapse.  
**Why it breaks:** Pete needs permits. Hank needs daily log. If those pages are hollow, the demo has a seven-minute dead zone.  
**Fix:** Port `useProjectWorkflowState` + `ProjectContextBanner` pattern from q2/q4/q5 into q8 and q15 (and q6, q11) following the formulaic template. Verify in `app/src/components/workflows/` and extend `src/lib/workflows.ts`.  
**Proof of readiness:** Existing template works cleanly on Estimating and Code Compliance.

---

### 2. **Jurisdiction context doesn't propagate from project to workflow** ⚠️ DEMO-RISK
**Impact:** 4 (Accuracy loss, multi-state fail) | **Effort:** S (5 lines)  
**Surface:** Code Compliance workflow defaults to "IBC 2024 (International)" even though project's San Diego context is in `raw_input` AND AI correctly named "Title 24"  
**The problem:** If Diana (CA Title 24) or Sarah (NYC) loads their project, the jurisdiction picker still reads "IBC generic." They'll assume the AI data is wrong. They're right to be skeptical.  
**Why it breaks:** Jurisdiction is the ONLY filter that makes code accuracy credible. If it defaults wrong, trust erodes.  
**Fix:** In `CodeComplianceClient.tsx` (or the equivalent in Project Spine v1), seed `jurisdictionId` from `project?.jurisdiction` on mount. If not set, infer from raw_input's location string (e.g., "San Diego, CA" → lookup CA in amendments data). ~5 lines in useEffect.  
**File path:** `app/src/components/workflows/CodeComplianceClient.tsx` or equivalent container.

---

### 3. **Stale empty-state copy doesn't update after project creation**
**Impact:** 3 (Confusion, "is this active?") | **Effort:** S (10 min, state management)  
**Surface:** `/killerapp?project=<id>` shows "You're not started yet. 7 stages to explore." even after a project is created and AI response is cached.  
**The problem:** Rico lands on the page with a live project but sees copy suggesting he hasn't started. Mental friction. Does his data persist? Is he in the right place?  
**Why it breaks:** Signal/noise. Copy contradicts reality. Drops confidence.  
**Fix:** Hydrate the empty-state component with `project?.id && project?.ai_summary` — if both exist, render "Working on: [job_description]" instead of "not started yet." File: `app/src/components/killerapp/KillerappShell.tsx` or the EmptyState child component.

---

### 4. **Workflow pages show "0 of N complete" on rehydrate**
**Impact:** 2 (Visual jank, not a blocker) | **Effort:** S (5 min per workflow)  
**Surface:** Progress counter on workflows doesn't seed from saved JSONB field; resets to 0/5 until user fills the first field.  
**The problem:** User comes back mid-workflow, sees "0 of 5" even though they've done step 2. Feels broken. Doesn't trust the save.  
**Why it breaks:** Weak adoption signal. If auto-save doesn't show visually, users re-do work.  
**Fix:** On workflow hydration, count the non-null fields in `workflow_state.fields` JSONB and seed the counter. File: `app/src/components/workflows/[workflowId]/ProgressBar.tsx` or equivalent.

---

### 5. **"You're confused" gap — no inline escape hatch for jargon**
**Impact:** 2 (Rage-quit on medium-jargon steps, 2+ personas) | **Effort:** M (2-3 hours, AI integration)  
**Surface:** Rico encounters "Conditional Lien Waiver" or "Pre-Bid Risk Score." No tooltip. No quick-explain button. Closes tab.  
**The problem:** Rico (08) and Jake (03) both need a way to ask "What does this mean?" WITHOUT leaving the step. Currently there's no glossary, no tooltip, no inline AI assist button.  
**Why it breaks:** Jargon kills adoption. Builders won't open a tool that makes them feel stupid.  
**Fix:** Add a glossary-button next to any unknown term (detect via a simple lookup against `data/glossary.json`). Clicking triggers a small inline tooltip (not a modal). If not in glossary, button routes to GlobalAiFab with context: "Explain '[term]' as it appears in [workflow/step] in 2 sentences."  
**File path:** `app/src/components/primitives/GlossaryButton.tsx` (new) + extend `data/glossary.json`.

## Three Handholding Patterns

### Pattern A: **Proactive AI Assist (Recovery from Dead Ends)**
**Where:** Any step showing an error, timeout, or missing data; also available as a quick-ask in collapsible footer  
**Implementation:**  
- Add a "Stuck?" button in the step footer that expands a mini-AI input (not a modal, in-flow).  
- User types: "Is this safe?" or "What should I do here?"  
- AI respond contextually using the current step, workflow, and project as context.  
- Response renders inline, doesn't interrupt the workflow.  
- If error (e.g., API timeout), auto-offer: "Try again?" button with one retry; if fails twice, show "Contact support" link with pre-filled context (workflow, step, error message).  
**Success metric:** Jake (JAKE-TC-02) can interrupt a code-compliance step mid-flow and ask "is this safe for Boston snow load?" without leaving the card.

---

### Pattern B: **Glossary Tooltips + Term Discovery**
**Where:** Contract templates, code responses, sequencing jargon, any undefined term  
**Implementation:**  
- Markup unknown terms with a `<GlossaryTerm term="Conditional Lien Waiver" />` wrapper.  
- Hovering shows a 2-3 sentence tooltip (CSS-driven, no modal).  
- Tooltip includes a "Learn more" link (optional) to a full glossary entry.  
- Terms auto-link to glossary; search the glossary by workflow stage (e.g., "Lock stage terms," "Build stage terms").  
- Glossary seeded with 40+ constructor-voice definitions; expandable via user contributions.  
**Success metric:** Rico encounters "Pre-Bid Risk Score," hovers, reads a plain-English definition, continues without closing the tab.

---

### Pattern C: **Visual Recovery from Workflow Dead Ends**
**Where:** Any step that results in an error, missing input, or "you can't proceed yet" condition  
**Implementation:**  
- Instead of a red error banner, use a card-based "here's what's missing" component that restates the problem in plain English.  
- Provide ONE clear next action: "Upload a photo to continue" or "Refill this field and try again."  
- If the step is legitimately stuck (e.g., API fails 3x, specialist prompt times out), offer "Contact support" with pre-populated issue context.  
- No error codes. No technical jargon. No "try again later."  
**Success metric:** Jake (JAKE-TC-11) hits an error mid-workflow, understands what went wrong in 10 seconds, and either fixes it or escalates with confidence.

## The First-Principles UX Bet

**Question:** What's the ONE design decision that, if made well, would prevent half the friction across all 10 personas?

**Answer: Project Spine visibility as the primary trust signal.**

The app's credibility depends on ONE moment: when the user submits a scope and sees their context travel alongside them across every workflow. If the project context (raw_input, AI take, location, jurisdiction) is **visible, persistent, and travels cleanly**, users trust the app is listening and remembering. If they click a second workflow and see a blank form, trust vanishes.

The fix is NOT just wiring the 14 unwired workflows — it's making the ProjectContextBanner a non-negotiable, always-on visual affordance. The banner should:
1. Render at the top of EVERY workflow page.  
2. Show: raw scope + AI summary + jurisdiction + timestamp of last edit.  
3. Be clickable to edit (jump back to /killerapp, edit the raw_input, AI re-analyzes).  
4. Serve as the source of truth for pre-fill (every field reads from this banner's context).  

If users see this banner on page load, they feel SEEN. Their context persists. They're not re-entering data. They're building ON something, not starting over. This single decision flips the narrative from "this app lost my data" to "this app remembers me."

**Justification (100 words):** Every persona's friction stems from a single root: feeling like the app doesn't understand their project. John loses $30k because there's no photo evidence. Rico rage-quits because jargon isn't explained. Maria doubts the AI because the jurisdiction defaulted wrong. Hank can't log fast because he has to re-enter crew names. All of these dissolve if the project context is visible and travels cleanly. Make the ProjectContextBanner permanent, always-on, and trustworthy — it becomes the app's source of truth and kills half the "feels nowhere" problem.
